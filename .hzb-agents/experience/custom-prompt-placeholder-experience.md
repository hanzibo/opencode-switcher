## 一、经验与教训总结

### 1.1 做得好的地方
- **高鲁棒性转义算法设计**：没有使用局限性大且有解析漏洞的正则负向后行断言，而是基于正则表达式匹配流（匹配 `\\\\` 或 `\${&}` 或 `${&}`），通过单个正则和回调函数，同时完成了“占位符精准替换”、“转义字符消除”与“非匹配降级追加”三合一功能，保证了解析的绝对正确。
- **完美的向后兼容性**：系统能精准判定是否有占位符替换发生。如果存在且发生替换，直接跳转；若提示词无占位符，自动执行追加逻辑，保证了用户存量旧数据的平滑过度。
- **敏捷单元测试前置**：在重新打包安装前，首先在沙盒中编写了涵盖所有边界条件的自动化测试用例（含转义符测试、多空值测试、连续重复占位符测试），测试通过后再在 GTK 中组装，避免了复杂的界面调试开销。

### 1.2 需要改进的地方
- **模块级常量声明意识需加强**：初始开发时将正则编译放到了 `_ask_custom_prompt` 触发逻辑内部，虽然单次点击开销不大，但确实产生了不必要的编译开销。代码审查阶段已将其优化提升为模块级常量。

## 二、关键问题与解决方案记录

### 问题1：如何正确区分转义占位符 `\${&}` 与转义反斜杠 `\\${&}`
- **问题描述**：
  1. 用户输入 `\${&}`，期望输出字面文本 `${&}`。
  2. 用户输入 `\\${&}`，期望输出字面文本 `\` 并替换占位符（即得到 `\原内容`）。
  普通正则替换很难合理区分这两个分支，容易产生误判或多余的反斜杠遗留。
- **原因分析**：这是文本模板引擎中经典的“转义字符状态机”问题。反斜杠自身也可以被反斜杠转义。
- **解决过程**：
  使用捕获组正则：`r'\\\\|\\(\${&})|(\${&})'` 并提供替换回调函数：
  - 如果匹配到 `\\\\`，保持为 `\\\\`（在处理后统一替换为单个 `\`）；
  - 如果匹配到组 1（`\${&}`），解开转义并返回 `${&}` 字符串；
  - 否则匹配到组 2（`${&}`），表示该占位符未被转义，设置 `has_placeholder = True` 并返回剪切板原内容。
- **最终方案**：
  在 [clipboard_panel.py](file:///home/hzb/opencode-switcher/clipboard_panel.py) 中，使用上述正则配合 `interpolated.replace('\\\\', '\\')` 进行最后的统一转义还原。
- **预防建议**：
  在设计任何带有“占位符+反斜杠转义”的文本插值逻辑时，优先考虑“匹配转义元字符、转义符号和普通占位符”的多级正则选择，避免单独用 lookbehind 正则，这样能自然涵盖复杂的转义连写。

## 三、技术要点沉淀

- **带转义的文本占位符插值标准模式**：
  ```python
  import re
  
  # 预编译正则模式
  PROMPT_PLACEHOLDER_RE = re.compile(r'\\\\|\\(\${&})|(\${&})')
  
  def interpolate(custom_prompt, original_content):
      has_placeholder = False
      def replace(match):
          nonlocal has_placeholder
          matched = match.group(0)
          if matched == '\\\\':
              return '\\\\'
          elif match.group(1):
              return match.group(1) # 还原为字面 ${&}
          else:
              has_placeholder = True
              return original_content # 替换为内容
      
      res = PROMPT_PLACEHOLDER_RE.sub(replace, custom_prompt)
      return res.replace('\\\\', '\\'), has_placeholder
  ```
- **GTK TextView 光标插入字符技巧**：
  在 `Gtk.TextView` 中直接调用 `buffer.insert_at_cursor(text)` 能够以最快、最安全（无需算坐标和索引）的方式在光标聚焦处插入文字。
